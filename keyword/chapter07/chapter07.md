낙관적 업데이트는 "성공할 확률이 매우 높고, 실패하더라도 되돌리는(Rollback) 비용이 적은 작업"에 가장 빛을 발합니다.

- **즉각적인 반응이 필요한 인터랙션**: '좋아요', '즐겨찾기 추가', '북마크' 등 사용자가 누르자마자 결과가 눈에 보여야 만족도가 올라가는 단순 토글 작업.
- **성공 확률이 압도적으로 높은 작업**: 네트워크 환경이 안정적이고 서버 측의 복잡한 유효성 검사가 필요 없는 단순 상태 변경.
- **반복적이고 빠른 입력이 필요한 기능**: '할 일 완료 체크', '리스트 순서 변경(Drag & Drop)' 등 사용자가 연속적으로 작업을 수행하는 경우.
- **단일 유저 중심의 데이터**: 다른 사용자와의 충돌 가능성이 적은 개인적인 데이터(나의 프로필 수정, 나의 메모장 등).

반대로 "정확성과 보안이 속도보다 훨씬 중요한 상황"에서는 절대 사용해서는 안 됩니다.

- **민감한 금융/결제 관련 작업**: 결제하기, 송금하기, 포인트 사용 등 실패했을 때 유저에게 금전적 손해나 혼란을 줄 수 있는 작업.
- **서버의 최종 승인이 필수적인 로직**: 중복 아이디 체크, 쿠폰 코드 적용 등 서버에서 DB를 대조해보고 나서야 성공 여부를 알 수 있는 복잡한 비즈니스 로직.
- **다음 동작이 서버 응답값에 의존할 때**: 게시글을 '생성'한 직후 서버에서 발급한 `ID`를 받아 바로 상세 페이지로 이동해야 하는 경우. (서버 응답 전에 가짜 ID로 이동하면 404 에러가 날 수 있음)
- **실패 시 롤백 UX가 너무 불쾌한 경우**: 유저가 공들여 작성한 긴 본문의 게시글 저장 등. 실패해서 쓴 내용이 갑자기 사라지면 유저는 큰 상실감을 느낍니다.

- **낙관적 업데이트(OptimisticUpdate)**를 왜 도입해야 하는지, 이 패턴이 해결하려는 문제를 실제 서비스 맥락에서 설명해보세요.
    
    **낙관적 업데이트가 해결하는 문제:**
    
    - **Perceived Performance (인지적 성능) 향상**: 실제 네트워크 속도는 같지만, 사용자가 느끼는 서비스 반응 속도를 혁신적으로 높입니다.
    - **로딩 스피너 제거**: 단순한 토글이나 짧은 텍스트 수정 작업에서 흐름을 끊는 스피너를 없애줍니다.
- TanStack Query 기반 구현 흐름을 `onMutate → (mutate) → onError → onSettled` 순서로 기술해주세요..
    
    
    | **단계** | **역할** | **주요 작업 내용** |
    | --- | --- | --- |
    | **`onMutate`** | **백업 및 선제 업데이트** | 1. 진행 중인 리페치 취소 (`cancelQueries`)
    2. 이전 데이터 스냅샷 백업
    3. 캐시 데이터를 새 값으로 즉시 수정 (`setQueryData`)
    4. **백업 데이터를 반환(context)** |
    | **`(mutate)`** | **실제 요청** | 서버에 데이터를 전송하는 비동기 함수 실행 |
    | **`onError`** | **복구(Rollback)** | 요청 실패 시, `onMutate`에서 반환했던 **context(백업본)를 사용하여 캐시를 이전 상태로 롤백** |
    | **`onSettled`** | **최종 동기화** | 성공/실패 여부와 상관없이 해당 쿼리를 무효화(`invalidateQueries`)하여 서버의 '진짜 데이터'와 동기화 |
- ToDo “생성” 및 “좋아요 토글”에 **낙관적 업데이트(OptimisticUpdate)**를 적용했을 때의 **실패/충돌 롤백 전략**을 설계해주세요.
    
    ### Case 1: 좋아요 토글 (Like Toggle)
    
    - **전략**: 단순 상태 반전 (Boolean Flip)
    - **낙관적 적용**: 현재 하트가 `off`라면 즉시 `on`으로 변경.
    - **실패 시 롤백**: 서버에서 에러(예: 401 권한 없음)가 나면, `onError`에서 보관 중이던 `previousData`를 꺼내 하트를 다시 `off`로 되돌립니다.
    - **핵심**: 유저는 하트가 잠깐 켜졌다가 다시 꺼지는 것을 보게 되며, 이때 에러 토스트(Toast)를 띄워 상황을 설명합니다.
    
    ### Case 2: ToDo 생성 (Create Todo)
    
    - **전략**: 임시 ID(Temp ID) 할당 및 제거
    - **낙관적 적용**: 서버 응답 전, `Date.now()`나 `crypto.randomUUID()`로 만든 **임시 ID**를 가진 가짜 ToDo 객체를 목록 맨 위에 즉시 추가합니다.
    - **실패 시 롤백**:
        1. `onError`에서 캐시 목록을 `onMutate` 시점의 백업 데이터로 통째로 덮어씌웁니다.
        2. 또는, 목록에서 해당 임시 ID를 가진 아이템만 `filter`로 제거합니다.
    - **주의 사항**: 생성된 할 일을 바로 수정하려고 할 때 서버가 준 진짜 ID가 없으면 에러가 날 수 있으므로, 롤백 전까지는 수정/삭제 버튼을 비활성화하거나 주의가 필요합니다.

**Q3 답변.**

### (a) 단순한 "빠름" 그 이상의 가치

모바일 환경은 생각보다 불안정합니다. 지하철이나 엘리베이터에서 네트워크가 잠시 끊겼을 때, 낙관적 업데이트가 없다면 유저는 버튼을 누를 때마다 멈춰있는 화면을 보며 **"응답이 없나?" 싶어 버튼을 여러 번 연타**하게 됩니다. 이는 서버에 불필요한 요청을 폭주시키고 유저를 불안하게 만들죠.
즉, 이 트릭은 유저에게 "네 명령은 잘 접수됐으니 안심하고 다음 일을 해!"라는 심리적 확신을 줍니다.

### (b) 실패와 햅틱(진동)의 조화

비행기 모드에서 체크가 됐다가 스르륵 풀리는 것이 바로 롤백(Rollback)입니다. 이때 툭- 하고 진동을 주는 이유는 "시각적 정보의 배신"을 알리기 위해서입니다.
유저는 방금 "체크가 됐다"고 눈으로 확인했습니다. 그런데 내 손가락이 닿지도 않았는데 화면이 스스로 바뀌면(체크 해제), 유저는 이를 인지하지 못하고 넘어갈 확률이 높습니다. 이때 **햅틱 피드백**을 주어 "방금 네가 한 동작이 서버 문제로 취소됐어"라는 신호를 촉각으로 강하게 전달하는 것이죠.

### (c) 모든 것에 적용하지 못하는 이유 (까다로운 지점)

왜 모든 곳에 쓰지 못할까요? 크게 세 가지 제약 때문입니다.

1. **서버 ID 의존성**: 메모를 작성하면 서버 DB에서 해당 메모리의 `id: 123`을 새로 만들어줍니다. 만약 낙관적으로 화면을 먼저 띄웠는데 유저가 그 즉시 '수정'을 누르면? 아직 클라이언트엔 ID가 없어서 서버에 "어떤 메모리를 고칠지" 말할 수 없습니다.
2. **데이터 관계의 복잡성**: "친구 수락"은 친구 목록뿐만 아니라 알림 목록, 채팅 목록 등 **여러 화면의 데이터에 동시에 영향**을 줍니다. 이 모든 걸 낙관적으로 처리했다가 한 번에 롤백하는 로직은 코드가 매우 복잡해지고 데이터가 꼬일 위험이 큽니다.
3. **돌이킬 수 없는 결정**: "친구 삭제"나 "회원 탈퇴" 같은 민감한 액션은 유저가 신중해야 합니다. 0.1초 만에 휙 사라지는 것보다, 서버의 확답을 받고 신중하게 처리되는 모습을 보여주는 것이 오히려 서비스에 대한 신뢰도를 높이기도 합니다.
