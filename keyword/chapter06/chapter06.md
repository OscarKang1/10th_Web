기존의 `cacheTime`이 v5에서 이름이 바뀐 것이 바로 `gcTime`입니다. **데이터가 사용되지 않을 때(inactive), 메모리에서 완전히 삭제하기 전까지 기다리는 시간**입니다.

- **동작**: 해당 데이터를 사용하는 모든 컴포넌트가 언마운트되어 '비활성(inactive)' 상태가 되면 타이머가 돌아갑니다. 이 시간이 지나면 가비지 컬렉터가 데이터를 메모리에서 지워버립니다.
- **기본값**: **5분** (300,000ms).
- **특징**: 데이터가 `stale` 하더라도 `gcTime`이 남아있다면, 다시 해당 화면으로 돌아왔을 때 **캐시된 데이터를 먼저 보여주고(UX 향상)** 그동안 백그라운드에서 최신 데이터를 받아옵니다.

`stale`은 '탁한', '신선하지 않은'이라는 뜻이죠. `staleTime`은 쿼리 결과가 **신선한(fresh) 상태로 유지되는 시간**을 말합니다.

- **동작**: 이 시간 동안은 페이지를 이동했다가 돌아오거나, 윈도우에 다시 포커스해도 **서버에 다시 요청(refetch)하지 않습니다.**
- **기본값**: **0** (즉, 데이터를 받자마자 바로 '신선하지 않은' 상태가 됩니다).
- **특징**: `staleTime`이 지나서 데이터가 `stale` 상태가 되면, 특정 조건(마운트, 포커스 등)이 충돌할 때 백그라운드에서 다시 데이터를 받아옵니다.

이 두 값을 어떻게 조합하느냐에 따라 사용자 경험이 천차만별로 달라집니다. 가장 효율적인 전략은 다음과 같습니다.

### 기본 원칙: gcTime > staleTime

항상 **`gcTime`은 `staleTime`보다 길게 설정**하는 것이 좋습니다. 만약 `staleTime`이 10분인데 `gcTime`이 5분이라면, 데이터가 아직 신선하다고 판단되어도 메모리에서 먼저 삭제되어 버리기 때문에 `staleTime` 설정이 무의미해집니다.

| **상황** | **추천 설정** | **이유** |
| --- | --- | --- |
| **자주 변하지 않는 데이터** (예: 유저 프로필, 설정값) | **staleTime ↑** (예: 5분~10분) | 불필요한 네트워크 요청을 줄여 서버 비용을 아끼고 속도를 높입니다. |
| **자주 변하는 실시간 데이터** (예: 주식 차트, 채팅 목록) | **staleTime: 0** | 항상 최신 상태를 유지해야 하므로 매번 다시 받아오게 합니다. |
| **사용자가 자주 왔다갔다 하는 화면** | **gcTime ↑** | 화면 이동 시 이전 데이터를 즉시 보여주어 로딩 스피너를 최소화합니다. |

### `invalidateQueries` 실행 시 일어나는 3단계 과정

1. **"이 데이터 상했어!" (Stale 처리)**
    - 지정한 `queryKey`(예: `['todos']`)에 해당하는 데이터를 즉시 **'신선하지 않은(Stale)'** 상태로 변경합니다. 캐시에 데이터는 남아있지만, 더 이상 믿을 수 없는 데이터라고 라벨을 붙이는 거예요.
2. **"당장 다시 가져와!" (Refetch 트리거)**
    - 만약 해당 데이터를 현재 화면에서 사용 중(Active)이라면, React Query는 별도의 명령 없이도 **백그라운드에서 즉시 서버에 새 데이터를 요청**합니다.
3. **"화면 업데이트" (UI Sync)**
    - 서버에서 따끈따끈한 새 데이터가 도착하면, 기존 캐시를 덮어쓰고 컴포넌트를 리렌더링합니다. 덕분에 사용자는 새로고침 없이도 방금 추가한 할 일이 목록에 나타나는 것을 보게 됩니다.
